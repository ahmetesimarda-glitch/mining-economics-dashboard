# DATABASE.md

> Database reference for the Mining Economics Dashboard. This document describes the actual persistence layer as defined in `prisma/schema.prisma`. It is written for senior engineers and AI coding assistants who need an accurate, repository-specific model of how data is stored, related, and mutated.

---

## 1. Overview

- **Engine:** PostgreSQL (see `docker-compose.yml`, which provisions `postgres:16-alpine`).
- **ORM:** Prisma `6.7.0` (`@prisma/client`), configured in `prisma/schema.prisma`.
- **Client generation:** the Prisma generator uses `binaryTargets = ["native", "linux-musl-arm64-openssl-3.0.x"]` so the client works both on local dev machines and on the Alpine/musl ARM64 container image built by the `Dockerfile`.
- **Connection:** the `DATABASE_URL` environment variable (see `.env.example`) is the single required connection string. The Prisma client is instantiated once through the singleton in `lib/prisma.ts` and imported everywhere as `@/lib/prisma`.
- **ID strategy:** every model uses a string primary key `id String @id @default(cuid())`. CUIDs are collision-resistant and generated by Prisma, so the application never assigns IDs manually.
- **Timestamps:** the root `MiningProject` model carries `createdAt DateTime @default(now())` and `updatedAt DateTime @updatedAt`. Child tables do not carry their own timestamps because they are always written as part of a project write.
- **Enums:** there are **no** Prisma `enum` types. All enum-like fields (for example `status`, `miningMethod`, `category`, `powerType`) are stored as `String` with an application-level default. Validation of allowed values happens in application code and UI, not in the database.

---

## 2. Entity-Relationship Overview

`MiningProject` is the root aggregate. Every other table is a child that belongs to exactly one project and is deleted with it.

```
MiningProject (1)
  ├── (N) CashFlowYear          // per-year projected cash flow rows
  ├── (N) CapexItem             // capital expenditure line items
  ├── (N) OpexItem              // operating expenditure line items
  ├── (N) Equipment             // machinery fleet with technical specs
  ├── (N) Personnel             // staffing roles and salaries
  ├── (N) ByProduct             // secondary revenue streams
  └── (N) MethodSpecificCost    // costs specific to the mining method
```

All relations are one-to-many from `MiningProject` to the child tables. There are no many-to-many relations and no self-references. Every foreign key is declared with `onDelete: Cascade`, so deleting a project removes all of its dependent rows in a single operation.

### Master Data (standalone roots)

`EquipmentCatalogItem` is **not** a child of `MiningProject`. It is the first Master Data catalog table: system reference equipment (manufacturer, model, technical and economic parameters). Projects copy (snapshot) values into `Equipment` rows via the project form; catalog edits never rewrite historical fleets. There is intentionally **no** foreign key from `Equipment` → `EquipmentCatalogItem` in this version.

### Demo analytics (standalone)

`DemoAnalyticsEvent` is a standalone event log (not related to `MiningProject` via FK). It stores pre-auth demo telemetry: `visitorId`, `eventType`, optional `projectId`, `path`, `metadata` (JSON), `createdAt`. Indexed on `visitorId`, `eventType`, `createdAt`, and `[visitorId, createdAt]`. After authentication, this becomes the basis for an Admin Dashboard (ownership-scoped).

## 3. Models

### 3.1 MiningProject

**Purpose:** the central aggregate root. It stores the full definition of a mining feasibility project: identification, mining method, financial assumptions, production parameters, depreciation and financing settings, and the cached results of the last economic analysis. Roughly 90 scalar fields plus seven relation fields.

**Primary key:** `id String @id @default(cuid())`.

**Indexes:** `@@index([status])` — supports filtering/listing projects by lifecycle state.

**Timestamps:** `createdAt` (`@default(now())`), `updatedAt` (`@updatedAt`).

**Field groups and notable defaults:**

- *Identification & lifecycle:* `name`, `description`, `location`, `commodity`, `miningMethod` (default `"openPit"`), `status` (default `"active"`), `currency` (default `"USD"`).
- *Time horizon & financial rates:* `projectLifeYears` Int default `30`, `discountRate` default `5.82`, `taxRate` default `22.0`, `royaltyRate` default `4.0`, `creditRate` default `4.0`, `creditYears` Int default `10`.
- *Currency handling:* `exchangeRate` default `1.0`, `manualExchangeRate` Bool default `false`.
- *Ore & production:* `oreGradeUnit` (default `"%"`), `unitPrice` default `75.0`, `annualProduction` default `2.0`, `productionUnit` (default `"Mt"`), `plantProcessingRate` default `35.0`.
- *Cost assumptions:* `contingencyRate` default `10.0`, `strippingUnitCost` default `1.05`.
- *Depreciation:* `equipmentDepLife` Int default `6`, `facilityDepLife` Int default `15`, `depreciationMethod` (default `"linear"`), `equipmentRenewalEnabled` Bool default `true`, `equipmentRenewalCycleYears` Int default `10`.
- *Capital structure:* `equityRatio` default `100`, `customCashFlows` Bool default `false`.
- *Cached analysis results (nullable Floats, written by the analysis engine):* `npv`, `irr`, `paybackPeriod`, `breakevenPrice`, `totalRevenue`, `totalCost`, `byProductRevenue`, `totalCapex`, `totalOpex`.

**Relations:** `cashFlows`, `capexItems`, `opexItems`, `equipments`, `personnels`, `byProducts`, `methodCosts` — all one-to-many, all cascade on delete.

**Business meaning of cached results:** the result columns are a denormalized snapshot of the last `performFullAnalysis` run (see `lib/calculations.ts`). They exist so the project list and detail views can render headline metrics (NPV, IRR, payback) without recomputing the full model on every read. They are recomputed and overwritten on every create/update.

### 3.2 CashFlowYear

**Purpose:** one row per project year holding the projected cash-flow breakdown. When `customCashFlows` is enabled on the parent project, these rows represent user-overridden values; otherwise they mirror the computed projection.

**Primary key:** `id` (cuid).

**Foreign key:** `projectId` references `MiningProject.id` with `onDelete: Cascade`.

**Constraints & indexes:** `@@unique([projectId, year])` guarantees at most one row per project per year; `@@index([projectId])` speeds up loading all years for a project.

**Columns:** `year` Int, and the following Floats all defaulting to `0`: `revenue`, `opex`, `depreciation`, `taxPayment`, `royalty`, `creditPayment`, `creditInterest`, `netCashFlow`, `cumulativeCashFlow`, `discountedCashFlow`.

**Business meaning:** mirrors the `CashFlowRow` shape produced by `calculateCashFlows` in `lib/calculations.ts`. The economic interpretation of each column (royalty basis, tax gating, discounting) is documented in `BUSINESS_RULES.md`.

---

### 3.3 CapexItem

**Purpose:** a single capital-expenditure line item (equipment purchase, facility, infrastructure, etc.).

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Columns:** `name` String, `category` String (default `"equipment"`), `quantity` Int (default `1`), `spareQuantity` Int (default `0`), `unitCost` Float, `totalCost` Float.

**Business meaning:** `totalCost` is a derived value. It is summed by `calculateTotalCapex` and contributes to the `MiningProject.totalCapex` cache and the Year-0 outflow.

---

### 3.4 OpexItem

**Purpose:** a single annual operating-expenditure line item.

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Columns:** `name` String, `category` String (default `"other"`), `annualCost` Float, `unit` String (default `"USD/yıl"`).

**Business meaning:** `annualCost` values are aggregated by `calculateTotalOpex` into the recurring operating cost used across every production year of the cash-flow model.

---

### 3.5 Equipment

**Purpose:** the machinery fleet. This is the richest child table, combining commercial data, general technical specifications, and category-specific attributes so the operational and emissions analyses can be driven from real fleet parameters.

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Core commercial columns:** `machineType`, `model`, `tonnageCapacity`, `quantity`, `spareQuantity`, `fuelConsumption`, `maintenanceCost`, `unitCost`, `totalCost`, `isCustom` (Bool — distinguishes user-added items from reference-catalog items).

**General technical columns:** `equipmentCategory` (default `"general"`), `dailyWorkHours` (default `8`), `maintenancePeriodHours` Int (default `500`), `operatorCount` Int (default `1`), `powerType` (default `"diesel"`), `hourlyFuelConsumption`, `productionImpact`.

**Category-specific columns (nullable, populated per equipment type):** `drillCapacity`, `holeDiameter`, `maxDrillDepth` (drilling); `bucketVolume`, `loadingCapacity` (loaders/excavators); `transportCapacity` (haulage); `crushingCapacity` (crushers); `gallerySuitability` (underground).

**Business meaning:** `totalCost` is derived as `(quantity + spareQuantity) * unitCost` at write time (see `equipFieldMap` in `app/api/projects/route.ts`). `powerType` drives emissions logic (diesel emits CO₂ at 2.68 kg/L; electric is excluded from direct emissions — see `BUSINESS_RULES.md`). Reference defaults come from `EQUIPMENT_REFERENCE` in `lib/market-reference.ts`.

---

### 3.6 Personnel

**Purpose:** staffing plan rows describing each role, headcount, and salary.

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Columns:** `role` String, `count` Int (default `1`), `monthlySalary` Float, `annualCost` Float, `isCustom` Bool.

**Business meaning:** `annualCost` is derived as `count * monthlySalary * 12` at write time (see `app/api/projects/route.ts`). Personnel annual cost feeds labor-cost aggregation and personnel-productivity analysis.

---

### 3.7 ByProduct

**Purpose:** secondary revenue streams (co-products / by-products recovered alongside the primary commodity).

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Columns:** `name` String, `annualProduction` Float, `productionUnit` String (default `"ton"`), `unitPrice` Float, `priceUnit` String (default `"USD/ton"`), `totalRevenue` Float.

**Business meaning:** `totalRevenue` is derived as `annualProduction * unitPrice` at write time. The sum of by-product revenue is cached on `MiningProject.byProductRevenue` and added to primary-commodity revenue in the cash-flow model.

---

### 3.8 MethodSpecificCost

**Purpose:** costs that only apply to a particular mining method (for example stripping-related or underground-specific costs) and do not fit the generic Capex/Opex tables.

**Primary key:** `id` (cuid). **Foreign key:** `projectId` to `MiningProject.id`, `onDelete: Cascade`. **Index:** `@@index([projectId])`.

**Columns:** `name` String, `category` String (default `"other"`), `value` Float, `unit` String (default `"MUSD"`).

**Business meaning:** on write, rows are filtered to require a non-empty `name` (see `app/api/projects/route.ts`) so blank UI rows are never persisted. Values are interpreted according to `unit` and folded into the relevant cost aggregates.

---

### 3.9 EquipmentCatalogItem (Master Data)

**Purpose:** system equipment catalog — the first Master Data entity. Stores manufacturer/model, category, description, optional `imageUrl`, technical specs (payload, bucket, power, weight, fuel L/h, fuel tank — nullable when unknown), economic params (list price with `isPriceEstimated`, useful life, availability, annual maintenance), `powerType`, metadata (`oemWebsite`, `country`, `notes`, `searchAliases`), optional `extraSpecs` JSON, and `isActive` / timestamps.

**Primary key:** `id` (cuid). **Unique:** `code` (stable seed key). **Indexes:** `category`, `isActive`, `manufacturer`, `model`.

**Not related to projects via FK.** The catalog UI and project form snapshot selected catalog rows into `Equipment` children. Catalog edits never rewrite historical fleets.

## 4. Data Lifecycle

1. **Create (POST `/api/projects`):** the request body is defensively defaulted field by field. Derived totals are computed in application code before any database write: `Equipment.totalCost`, `Personnel.annualCost`, `ByProduct.totalRevenue`, then `calculateTotalCapex`, `calculateTotalOpex`, and `performFullAnalysis`. `MethodSpecificCost` rows without a `name` are dropped. Everything is written in a single `prisma.miningProject.create` with **nested `create`** for all child collections, so a project and all its children are inserted atomically.
2. **Read (GET `/api/projects`):** returns projects `orderBy: { updatedAt: 'desc' }` and `include`s the child collections. On any error the handler returns an empty array with HTTP 200 (graceful degradation) rather than a 500, so the dashboard never hard-fails on listing.
3. **Read one (GET `/api/projects/[id]`):** loads a single project with its relations; returns 404 with `{ error: 'Proje bulunamadı' }` when missing.
4. **Update (PUT/PATCH `/api/projects/[id]`):** child collections are replaced using a **delete-and-recreate** strategy inside the update — existing child rows are cleared and rewritten from the incoming payload, and cached result columns are recomputed. This keeps the persisted state consistent with a fresh analysis rather than attempting per-row diffs.
5. **Delete (DELETE `/api/projects/[id]`):** deleting the `MiningProject` cascades to every child table via `onDelete: Cascade`; no manual child cleanup is required.
6. **Seeding:** `scripts/seed.ts` (and the safer `scripts/safe-seed.ts`, wired to `prisma db seed` via `tsx --require dotenv/config scripts/safe-seed.ts`) populate demonstration projects and the Equipment Catalog. Seeds are **idempotent upserts only** — never `delete` / `deleteMany`. Equipment Catalog rows upsert by stable `code` via `seedEquipmentCatalogIdempotent` (user-created catalog codes are left untouched). Demo projects upsert by fixed `seed-*` ids: create with nested children if missing, otherwise update scalars only (child fleets are preserved).

## 5. Query Patterns & Performance Considerations

- **Singleton client:** always import the shared client from `@/lib/prisma`. It is cached on `globalThis` outside production to avoid exhausting connections during hot-reload. Never construct a new `PrismaClient` per request.
- **Status filtering:** listing/filtering by `status` is backed by `@@index([status])` on `MiningProject`.
- **Child fan-out:** `include`-ing all seven child collections on a list query multiplies the row count returned. For large datasets prefer selecting only the scalar columns you need (the AI-analysis route already uses a scalar-only `select` for its benchmark query) instead of full `include`.
- **Write amplification:** the update path deletes and recreates child rows on every save. This is simple and correct but writes O(number of children) rows per save; keep payload sizes reasonable and batch updates client-side.
- **Compute-on-write, not on-read:** headline metrics are computed once during the write and cached on `MiningProject`, so reads stay cheap. The heavy analysis (including a 2000-iteration Monte Carlo) runs synchronously during the request that triggers it — keep that in mind before adding more per-request recomputation.
- **Ephemeral connections:** treat database connections as short-lived. Do not rely on session-level state, long-lived transactions, or persistent temp tables.

## 6. Future Expansion Notes

- **Master Data catalogs:** `EquipmentCatalogItem` is the first implemented catalog. Future catalogs (commodity, country, currency, units, fuel types, processing methods, labour categories) should follow the same standalone-root pattern — no FK from projects into catalog rows for live joins. Projects **snapshot** values at write time.
- **Authentication tables:** `next-auth` and `@next-auth/prisma-adapter` are installed but there are currently **no** `User`/`Account`/`Session`/`VerificationToken` models in the schema. Adding auth means introducing those models (and a `MiningProject.userId` owner relation) as an additive, non-destructive migration. Migrate `/internal/demo-analytics` → Admin Dashboard with ownership filters; keep `DemoAnalyticsEvent` additive.
- **Mining news / AI brief:** `lib/news/` is placeholder-only. Live commodity/news APIs and AI daily summaries should implement `NewsService` without changing `NewsCard` / dashboard contracts. Optional additive `NewsArticle` persistence later.
- **GIS:** location remains free-text `location` + lat/lng on `MiningProject`. Future GIS can add normalized country/state/city columns or a `ProjectLocation` child additively; Nominatim search already returns normalized parts.
- **Introducing enums:** enum-like `String` fields could be promoted to Prisma `enum` types for stronger typing, but doing so is a breaking migration and requires backfilling/validating existing values first.
- **Auditing & soft-delete:** child tables have no timestamps and deletes are hard cascades. If history or soft-delete is needed, add `deletedAt`/audit columns additively rather than changing cascade behavior.
- **Migration safety:** all schema changes must be backward-compatible. Never run `prisma db push --accept-data-loss` or `--force-reset` against the shared database; prefer additive migrations and explicit backfills.
